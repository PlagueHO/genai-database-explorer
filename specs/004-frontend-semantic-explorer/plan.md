# Implementation Plan: Frontend Semantic Model Explorer

**Branch**: `004-frontend-semantic-explorer` | **Date**: 2026-02-24 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-frontend-semantic-explorer/spec.md`

## Summary

Build a React single-page application for browsing and editing semantic models generated by GenAI Database Explorer. The frontend integrates into the .NET Aspire AppHost via `AddViteApp()`, communicates with the existing REST API, and provides paginated entity lists, detail views with inline editing, and a placeholder chat panel for future agentic capabilities. The technical approach uses React 19 + Vite + TypeScript with FluentUI React Components v9 for the design system, Tailwind CSS v4 for layout utilities, TanStack Query v5 for server state, and React Router v7 for navigation.

## Technical Context

**Language/Version**: TypeScript 5.x targeting ES2022, React 19, Node.js 22 LTS
**Primary Dependencies**: React 19, Vite 6, @fluentui/react-components v9, tailwindcss v4, @tanstack/react-query v5, react-router v7
**Storage**: N/A (frontend consumes REST API; backend handles persistence)
**Testing**: Vitest (unit), React Testing Library (component), Playwright (E2E)
**Target Platform**: Modern browsers (Chromium, Firefox, Safari), served via Vite dev server in Aspire
**Project Type**: Web application (SPA frontend + existing .NET backend)
**Performance Goals**: Dashboard load < 3s, entity detail load < 2s, search response < 200ms client-side filtering (SC-001, SC-002, SC-003)
**Constraints**: Must integrate into existing Aspire AppHost; must use existing API endpoints; backend API extensions needed for NotUsed editing and column-level PATCH
**Scale/Scope**: Up to 500 tables (SC-007), 7 pages, ~25 components, 12+ API endpoints

## Complexity Tracking

> Constitution principle V mandates MSTest+FluentAssertions+Moq for unit tests.
> This is a .NET-specific requirement. The frontend uses Vitest+RTL+Playwright
> as the equivalent stack for React/TypeScript. This is not a violation but a
> technology-appropriate adaptation.

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Vitest instead of MSTest | Frontend is TypeScript/React — MSTest does not apply to JavaScript | No alternative — MSTest cannot test React components |
| New top-level directory (`genai-database-explorer-frontend/`) | Frontend is a separate Node.js project with its own dependencies, build tools, and runtime | Nesting inside `genai-database-explorer-service/` mixes .NET and Node.js toolchains, complicates builds, and confuses dependency resolution |
| TanStack Query instead of custom state | Server state caching, loading/error states, and query invalidation are complex to implement correctly | Custom fetch + useState would be simpler but lacks caching, deduplication, and optimistic updates needed for editing workflows |
